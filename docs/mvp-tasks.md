# MVP Task List

Tasks to reach MVP (PRD Milestones 1–5). Each task is scoped for a single agent session. Tasks are ordered by dependency — later tasks depend on earlier ones.

The guiding principle: **one museum end-to-end before adding complexity.** The Met is the vertical slice that validates the entire pipeline. Brooklyn and Harvard stress-test normalization. Authority data is built from real ingested data, not guesses.

Phase-0-source handoffs live at `docs/handoff-<source>-*.md` (e.g. `docs/handoff-dodson-hilton-next-chunk.md`, `docs/handoff-phase-0-transcription.md`) and persist across sessions. Ad-hoc session handoffs may also be written at `docs/handoff-session-*.md` when a session ends with significant transient state; these are deleted by the next session that consumes them.

#### MVP "good enough" breakpoint — Phase-0 done criteria

Phase-0 source acquisition is considered **MVP-complete** and unblocks the start of Phase A (authority curation → enrich assets → beta launch) when **all of the following are true**:

**Must-land before Phase A can start (MVP-blocking):**

0. **Schema-integrity gate.** A source does not count as `✅` complete for Phase-A purposes while its **schema** — the typed fields in its `reconciled.jsonl` — carries an unresolved **P1 schema-design defect**. Three specific classes qualify: (a) **field-level rule-4 violation** — a single typed field encoding two unrelated concepts (e.g., the Leprohon `stage_suffix` overload covering both same-king titulary stages and distinct queen-consort sub-entries); (b) **schema-level rule-3 violation** — a field invariant, type constraint, or cross-field contract that exists only in README / prompt / `transcribe.md` prose with no enforcing test (not "a test is missing somewhere", but specifically a test that would enforce a field-level semantic the schema claims to hold); (c) a **typed field shape that demonstrably cannot represent the data Leprohon/Baud/etc. prints faithfully** — forces the extractor into a silent lossy encoding. A source hitting any of those three must be flagged `🟡 provisionally complete pending schema PR` with the open schema work recorded on its row; "substantively complete" is NOT a synonym and must not be used to cover a known schema-design defect. The schema PR either ships first or the source holds the provisional flag until it does.

   **Out of scope for this gate:** content-level P1s do NOT demote `✅` under this gate. That includes parser / merge / `fix_rows` bugs that produce wrong values in correctly-shaped fields (pharaoh.se Roman date signs, Caracalla index-vs-page shift, Baud's missed eldest-son role-derivation rule), merge-logic defects that admit arbitrary rows (Ryholt's no-majority singleton admission), and Rule-1 provenance gaps (Kitchen's uncommitted source artifacts). These are real P1s — several are named in the sweep audit — but they are **tracked separately** under the **Known blind-spot audits** section below as pre-Phase-A blockers; they are fixable within a source without changing its schema, and they do not compound through Phase-A consumers the way schema debt does.

   **Rationale.** Schema debt on a primary authority compounds through Phase-A consumers in a way content bugs don't. A wrong Caracalla date in pharaoh.se is one fix in one source; a `stage_suffix` overload propagates into the shape of `rulers.json` and every downstream query that reads it. The cost curve of schema fixes flips from "hours" at chunk-author time to "weeks + coordination" once `rulers.json` / `sites.json` start consuming the field. The 2026-04-23 Leprohon `stage_suffix` overload is the anchoring case (see post-mortem in session log).

### 3.2 Phase A: Authority curation

Hand-curate the four authority files in dependency order. Each file has the mandatory `_source` block (ADR-012), uses the structured shapes defined in ADRs 013 and 016, and seeds its `aliases` array from the most-frequent distinct raw values in `catalog.artifacts`.

1. **`dynasties.json`** — Dynasties 0–31 + Ptolemaic. Each entry: `id`, `display`, `dates`, `parent_period`, `polity` (for concurrent Intermediate-Period dynasties — 13th, 14th, 15th, 16th, 17th in different regions), `concurrent_with`, `aliases`
2. **`periods.json`** — Predynastic through Ptolemaic. Sub-periods (Amarna, Ramesside, Saite) are their own entries with `parent_id`. Sub-period replaces parent when raw text resolves more specifically
3. **`rulers.json`** — Per ADR-016: canonical `display` (Conventional English Display Form), structured `titulary` object with all five name parts, flat `aliases` for the matcher. Coverage priority: New Kingdom and Late Period
4. **`sites.json`** — Hierarchical structure (`egypt > upper_egypt > thebes > deir_el_bahri > tt_358`), ~100 most-referenced sites + KV/TT tombs
