# Enrich stage + Hapi Brain — implementation plan

## Context

**What Hapi Brain is.** A read-only, human-inspectable view of the normalization/authority layer — the UI side of "show your work." For every canonical entity (ruler, site, period, dynasty) it renders the artifacts resolved to it, the raw-museum-string → authority-entry evidence chain for each resolution, and a pointer to the source authority file on disk. It is **not** a wiki; its differentiator vs. Wikipedia / Trismegistos / Pharaoh.se / TLA is *"cross-museum artifact corpus cross-referenced against the normalization decisions that produced it, with the reasoning visible."* v0 is strictly inspection — no editing, no flagging UI. Primary audience: project owner debugging the pipeline + scholars verifying provenance. Feedback happens out-of-band (email).

**Why now.** The enrich stage is a blank slate (`pipeline/pipeline/assets/enrich/__init__.py` is 0 lines). The `catalog.artifacts` table already has output slots (`ruler_id`, `origin_site_id`, `origin_site_display_name`, `tomb_temple_id`) sitting empty because no resolver exists yet. Because we're writing the enrich stage from scratch, we can bake evidence-recording in as a first-class output rather than retrofitting it later. The alternative — enrich writes only the resolved ID to `artifacts` — would erase the reasoning trail and make Brain impossible to build without a second pass through the whole schema.

**Intended outcome.** Every artifact field that should trace to an authority (`ruler`, `site`, `tomb_temple`, later `period`/`dynasty`) gets resolved deterministically against an authority source, with the current decision — exact/fuzzy/unmatched, the alternatives considered, the source authority file — stored as a single authoritative row, and all decision changes preserved in a separate append-only history table. A read-only Next.js UI under `/brain/*` renders this data as entity-centric provenance pages.

## Architecture decisions

### 1. Two-table model: `entity_resolutions` (current) + `entity_resolution_history` (audit)

ADR-009 defined `catalog.fuzzy_match_reviews` as an ambiguous-match queue. Replace it with a two-table model — the review-queue semantics become a filter over the current table. **This amends ADR-009**; per rule 10 (no backwards compatibility) the drop is direct, no view alias, no shim. `fuzzy_match_reviews` is empty today (no producer has ever existed), so the migration is an explicit `drop_table` + `create_table` — not an alembic autogenerated rename, which would lose data if the table had any.

`catalog.entity_resolutions` — exactly one row per `(artifact_id, field)` (`UNIQUE` constraint). Enrich re-runs UPSERT via `INSERT ... ON CONFLICT (artifact_id, field) DO UPDATE`. This is the current state; all reads hit only this table, no flag filter.

```
id, artifact_id, field, raw_value, raw_value_hash,
matched_id, matched_display_name,
match_type ('exact'|'fuzzy'|'unmatched'),
score (nullable),
alternatives (JSONB, nullable),
source_authority_file, authority_source_version (content hash of JSONL at resolution time),
resolver_name, resolver_version,
resolved_at,
review_status (nullable), reviewed_by, review_notes, reviewed_at
```

`catalog.entity_resolution_history` — append-only. A row is written **only when the decision actually changes** (different `matched_id`, `match_type`, or `authority_source_version` from what's already in `entity_resolutions`). Carries the same columns plus `changed_at`. A no-op re-run writes nothing; this is the "dedup guard" falling out structurally rather than as a bolted-on skip.

- **`field` enum** — `ruler`, `site`, `tomb_temple`, `period`, `dynasty`. Keep `site` and `tomb_temple` separate (different authorities). Only `ruler` has a producer in PR1; `site` in PR2; the rest are reserved for later PRs.
- **`match_type`** — exactly three values in PR1–PR3: `exact`, `fuzzy`, `unmatched`. No `manual` until a human-override workflow exists.
- **`alternatives`** — record when (a) match_type='fuzzy', (b) match_type='exact' with >1 candidate (ambiguous exact), or (c) top-2 fuzzy gap is small. Do NOT populate for unique exact matches (noise).
- **`review_status`, `reviewed_by`, `review_notes`, `reviewed_at`** — added in PR1, always NULL until PR2 introduces fuzzy matching. Forward-compatible.

### 2. Authority in Postgres — fat JSONB projection (web reads Postgres only)

JSONL files in `pipeline/pipeline/authority/sources/` remain the **source of truth** for the pipeline (constitutional rule 4). Pipeline loader assets project them into Postgres tables that carry both the scalar fields the resolver needs AND the full source row as JSONB, so the web app queries Postgres exclusively — no filesystem access at runtime. This avoids a sharp problem: Next.js production deploys (Vercel, Fly, containers) don't ship sibling repo directories, so any design that asks the web server to `fs.readFile` JSONL files would break deployment.

- `catalog.authority_rulers` — `id` (pharaoh-se slug), `display_name`, `dynasty_label`, `source_file` (relative path), `source_row_hash` (content hash of the JSONL row), `raw_row` (JSONB of the full pharaoh-se row — prenomen, nomen, horus/nebty/golden-horus/throne/birth names arrays, chronologies, everything). Brain UI reads `raw_row` to render source-specific fields.
- `catalog.authority_ruler_variants` — **separate variant index** so multiple sources can contribute variants to one canonical ruler without schema migration: `ruler_id`, `variant_string` (as-written), `normalized` (canonicalized for matching), `source_file`, `is_variant` (boolean — flags entries where the source marked `is_variant=true`, so Brain can distinguish primary attestation from variant). v0 populates from pharaoh-se only; Leprohon / Dodson-Hilton / Baud / Ryholt plug in later as rows, not schema changes.
- `catalog.authority_sites` — equivalent shape for iDAI (PR2): id, display_name, parent_id, coordinates, source_file, source_row_hash, raw_row.

**Source of truth, explicitly:** JSONL is canonical for the *pipeline* (what the resolver reads to build the DB tables). Postgres is the canonical runtime read path — no other component reads JSONL directly. The DB is a deterministic, hash-verified projection of the JSONL. This is rule 4 compliant: one source upstream, one source at runtime, a verified reproducible transformation between them.

### 3. Shared normalization callable

The loader writes `authority_ruler_variants.normalized` by canonicalizing each variant string (lowercase + Unicode NFKC + collapse whitespace + strip punctuation). The resolver must apply the **same** canonicalization to its query string before lookup — otherwise exact matches silently miss when the function drifts.

Pin to a single module: `pipeline/pipeline/authority/normalize.py` exports one function `canonicalize_variant(s: str) -> str`. Both the loader and the resolver import it. A structural test asserts both call sites (`authority/loaders/pharaoh_se.py` and `authority/resolver.py`) import from this module — no inlined duplicates.

### 4. Resolver — two-stage per ADR-009

`pipeline/pipeline/authority/resolver.py`:
1. **Exact stage** — `canonicalize_variant(raw_value)` lookup against `authority_ruler_variants.normalized`. 1 hit → `match_type='exact'`, populate `artifacts.ruler_id` and overwrite `artifacts.ruler_display_name` with `authority_rulers.display_name` (see rule below). >1 hit → `match_type='exact'`, all candidates in `alternatives`, leave `artifacts.ruler_id` NULL.
2. **Fuzzy stage** (PR2) — rapidfuzz `token_sort_ratio` over all normalized variants; top match → `match_type='fuzzy'`, top-5 in `alternatives`, `review_status='pending'`. **Never auto-accepts** (per ADR-009); leaves `artifacts.ruler_id` NULL.
3. **Unmatched** — `match_type='unmatched'`, `matched_id` NULL, artifacts row untouched.

**`artifacts.ruler_display_name` overwrite policy:** when `ruler_id` is successfully populated (exact-unique only in PR1–PR2), overwrite `ruler_display_name` with the canonical `authority_rulers.display_name`. This keeps search facets clean — "Tuthmosis III" and "Thutmose III" aggregate to the same facet value once resolved. The museum's original raw string is preserved in `entity_resolutions.raw_value` (rule 6: raw data is sacred). When `ruler_id` is NULL (ambiguous, fuzzy-pending, unmatched), leave `ruler_display_name` as the museum's text.

**Zero `try/except` in resolver or loaders.** Raise on malformed JSONL, on schema violations, on unexpected structure. Match the existing mapper convention (rule 2).

### 5. Three-PR split

Full scope is too large for one PR (schema change + migration + 3 loader/resolver modules + 4 Dagster assets + 6 web routes + query layer + introspection). Split into three independently mergeable slices. Each demoable on its own; PR1 proves the spine works even without UI.

### 6. Enrich-asset dependencies — no hardcoded museum list

`pipeline/CLAUDE.md` says *"the enrichment and indexing stages operate on all normalized data regardless of source."* Do not hardcode `deps=["normalize_met", "normalize_harvard", "normalize_brooklyn", ...]` in enrich assets — every new museum would become a churn point and the third PR to add a museum will forget to update it.

Use a Dagster `AssetSelection.groups("normalize")` dependency (or equivalent — a `normalize_all` sentinel asset that fans in from every `normalize_*`), so enrich automatically picks up every new museum. Add a structural test to `tests/test_structure.py` that asserts enrich's resolved deps cover every registered museum's normalize asset — a rule-3 enforcement check.

**Also audit `sync_search` for the same bug.** If it has a hardcoded list, fix it in PR1 as part of the enrich work (don't copy the bug into enrich). If it uses a group selector already, copy that pattern.

- `pipeline/pipeline/authority/loaders/pharaoh_se.py` (new) — parse `pipeline/pipeline/authority/sources/pharaoh-se/reconciled.jsonl`. **Correct extraction shape** (verified against real rows): top-level `slug` (→ ruler id), `display` (→ display_name), `alt_labels` (flat string array → variants), `dynasty_label`; then descend into each `*_names` array (`birth_names`, `throne_names`, `horus_names`, `nebty_names`, `golden_horus_names`, `prenomen`, `nomen` when present) — each element is an object `{name, transliteration, translation, gardiner, is_variant, sources}`; pull `.name` from each object as a variant, and carry the `is_variant` flag into `authority_ruler_variants.is_variant`. - `pipeline/pipeline/assets/enrich/__init__.py` — currently empty.

## PR3 — Hapi Brain UI

### Routes (under `web/src/app/brain/`)

- `/brain` — entity-type tiles (rulers, sites).
- `/brain/rulers` — list (from `catalog.authority_rulers` joined with artifact counts).
- `/brain/rulers/[id]` — ruler detail: renders source-specific fields from `authority_rulers.raw_row` (prenomen, Gardiner codes, titulary), `_source` card, artifacts grid (reusing `web/src/components/artifact-card.tsx`), `EvidenceChainTable` (raw → matched with match_type/score/alternatives), aggregated origin-site list (`GROUP BY` on the fetched artifact set).
- `/brain/sites` + `/brain/sites/[id]` — analogous for sites (using `authority_sites.raw_row`).
- `/brain/unresolved` — **debugging surface**: filterable list of `entity_resolutions` rows where `match_type in ('fuzzy','unmatched')`, grouped by field + museum. Read-only.
- `/brain/museums/[museum]` — resolution-rate tiles per field (exact/fuzzy/unmatched counts).
- `/brain/authority/[source_file]` — **rule-1 traceability page**: renders the `_source` block from any row in `authority_rulers`/`authority_sites` (citation, retrieval date, license, raw file pointer), row count, last-loaded timestamp.

### Critical files

- `web/src/lib/queries/entities.ts` (new) — Drizzle-based Postgres queries: `getRulerById`, `getArtifactsByRuler`, `getResolutionsForRuler`, `getUnresolved`, etc. All queries hit Postgres only; no filesystem reads.
- `web/src/components/evidence-chain-table.tsx` (new).
- `web/src/components/entity-header.tsx` (new).
- `web/src/lib/db/entity-resolutions.ts` (new) — typed wrappers over the JSONB `alternatives` and `raw_row` columns, cast at the query boundary (do not hand-edit introspected `schema.ts`, per rule 9).
- Reuse `web/src/components/artifact-card.tsx` for artifact grids.
- Reuse `web/src/components/artifact-image.tsx` license-check logic (rule 8).

### Typesense

Bypass entirely — Brain pages query Postgres directly. Typesense stays for the user-facing search UI at `/`.

## Verification

### Pipeline (PR1)

```bash
# Migration (creates entity_resolutions, entity_resolution_history, authority_rulers, authority_ruler_variants; drops empty fuzzy_match_reviews)
cd pipeline && uv run alembic upgrade head

# Load authority (populates authority_rulers + authority_ruler_variants from pharaoh-se JSONL)
cd pipeline && uv run dagster asset materialize -m pipeline.definitions --select load_authority_rulers

# Run enrich (reads artifacts, writes entity_resolutions, updates artifacts.ruler_id for exact-unique)
cd pipeline && uv run dagster asset materialize -m pipeline.definitions --select enrich_rulers

# Test suite (value assertions + golden-file snapshot + hashes_snapshot + AST import check)
cd pipeline && uv run pytest

# Data inspection — primary demo surface for PR1
psql $DATABASE_URL -c "
  SELECT field, match_type, COUNT(*)
  FROM catalog.entity_resolutions
  GROUP BY field, match_type;
"
psql $DATABASE_URL -c "
  SELECT a.id, a.ruler_display_name, a.ruler_id,
         e.match_type, e.source_authority_file, e.alternatives
  FROM catalog.artifacts a
  LEFT JOIN catalog.entity_resolutions e
    ON e.artifact_id = a.id AND e.field = 'ruler'
  WHERE a.ruler_display_name IS NOT NULL
  LIMIT 20;
"
```

Note: the psql queries above reference columns introduced by the PR1 migration; run `alembic upgrade head` first. No `is_current` filter is needed because `entity_resolutions` holds exactly one row per `(artifact_id, field)`.

PR1 is demoable here: the SQL *is* the provenance explorer. UI is skin over this.

### Constitutional-rule audit (every PR)

- Rule 1: every `entity_resolutions` row has `source_authority_file` + `authority_source_version`; every `authority_rulers` row has `source_file` + `source_row_hash` + full `raw_row`.
- Rule 2: resolver + loaders contain zero `try/except`; raise on malformed data.
- Rule 3: `test_hashes_snapshot` + Dagster asset check + AST import check + structural tests all enforce invariants in CI.
- Rule 4: JSONL is source of truth for the pipeline; Postgres is source of truth at runtime; DB is a hash-verified deterministic projection of JSONL. No runtime filesystem reads from web.
- Rule 5: fixture tests assert all populated fields + golden-file snapshot of `entity_resolutions`.
- Rule 9: `web/src/lib/db/schema.ts` regenerated only via `drizzle-kit introspect`; typed JSONB wrappers in sibling files.
- Rule 10: table replacement is direct (drop + create); no compatibility view.
- Rule 11: hashes-snapshot test runs in the existing pytest CI job with no DB dependency; live-DB parity runs as a Dagster asset check in the integration job.
